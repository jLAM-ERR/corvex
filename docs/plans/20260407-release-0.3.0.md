# Release 0.3.0 — Bug Fixes, AmneziaWG, Windows Support, Static Port

## Overview

Release 0.3.0 addresses three bugs and adds four features to the corvex CLI:

**Bugs:**
- B1: Ensure all required directories exist on `corvex start` (config, log dirs)
- B2: Add missing corporate-dns routing rule (port 53) to xray config.json after DNS sync
- B3: Propagate `log.xray.loglevel` from corvex.json into xray config.json on both create and update paths

**Features:**
- F1: Static proxy port via `proxy.port` in corvex.json (required field, no random fallback)
- F2: AmneziaWG support as alternative tunnel engine (`vpn://` URI), with xray as local routing layer
- F3: Windows corporate DNS discovery via `GetAdaptersAddresses` WinAPI + registry NRPT rules
- F4: Windows build with platform abstraction for proxy, network, DNS, and process management

**Acceptance Criteria:**
- All existing `cargo test` tests continue to pass
- Each bug fix has regression tests
- `corvex start` creates missing directories gracefully (including permission-denied paths)
- Xray config.json contains corporate-dns routing rule with port 53 after DNS sync
- Log level from corvex.json propagates to xray config on both fresh create and update
- Static port from corvex.json `proxy.port` is required and used; error if not set
- `vpn://` URI is parsed, AWG tunnel starts, xray runs as routing layer
- Project compiles for Windows target (`cargo check --target x86_64-pc-windows-msvc`)
- `cargo clippy` and `cargo fmt --check` pass

## Context (from discovery)

**Files/components involved:**
- `src/main.rs` — CLI routing, `cmd_start`, `build_xray_log_config`, `main_algorithm`, `update_config_port`
- `src/config.rs` — `Config` struct with XDG paths
- `src/settings.rs` — `CorvexSettings`, `RoutesSettings`, `LogSettings` structs
- `src/protocol.rs` — `parse_uri`, `create_config`, `apply_to_config`, `XrayLogConfig`
- `src/dns.rs` — `parse_scutil_dns`, `sync_to_config`, `init`
- `src/port.rs` — `find_free_port`
- `src/proxy.rs` — macOS proxy enable/disable/status via `networksetup`
- `src/network.rs` — macOS `detect_active_service` via `route`/`networksetup`
- `src/xray.rs` — xray process lifecycle (start/stop/reload) using `nix` signals
- `src/traffic.rs` — routing rule builder
- `src/health.rs` — server health checks (uses `std::os::unix::fs::OpenOptionsExt` — needs cfg-gate for Windows)
- `src/subscription.rs` — subscription download and decode

**Key architectural decisions:**
- Engine mode enum (`EngineMode::Xray` / `EngineMode::Awg`) with match-based dispatch — simpler than trait for 2 engines
- AWG mode: AWG creates tunnel via `awg-quick up`, xray runs as local routing layer (proxy outbound = `freedom`)
- Platform abstraction: cfg-gated module re-exports with concrete types (no dynamic dispatch) — `#[cfg(target_os = "macos")] pub type PlatformImpl = MacOsPlatform;`
- Windows DNS: dual-source — `GetAdaptersAddresses` WinAPI + registry NRPT rules, no PowerShell subprocess

**Dependencies identified:**
- New: `windows-sys` crate (cfg-gated for Windows)
- Existing `nix` crate becomes cfg-gated for unix only

## Agent Parallelization Strategy

Tasks are sequenced to avoid file conflicts between parallel agents:

- **Phase 1** (sequential — B1 and B3 both touch main.rs): B2 parallel with (B1 then B3)
  - B1 touches: `main.rs` (add `ensure_directories()` at top of `cmd_start`)
  - B2 touches: `dns.rs` only (add routing rule in `sync_to_config`)
  - B3 touches: `protocol.rs` (`apply_to_config` log update) + `main.rs` (call site update in `cmd_start` update path, lines 185-199)
  - B2 can run in parallel with B1+B3 since it only touches `dns.rs`

- **Phase 2**: F1 (static port)
  - Touches: `settings.rs` (new struct), `main.rs` (port selection logic)

- **Phase 3**: F4 (platform refactor) — Tasks 5-7
  - Moves code from `proxy.rs`, `network.rs`, `dns.rs` → `platform/` module
  - Updates `main.rs`, `xray.rs`, `health.rs`, `config.rs`, `Cargo.toml`

- **Phase 4**: F2 (AmneziaWG) — Tasks 8-11
  - New `engine/` module, updates to `main.rs`, `settings.rs`, `Cargo.toml`
  - Depends on platform abstraction from Phase 3

- **Phase 5**: F3 (Windows DNS) — Task 12
  - Implements `platform/windows.rs` DNS discovery
  - Depends on platform abstraction from Phase 3

## Technical Details

### corvex.json additions

```jsonc
{
  "uri": "vless://... OR vpn://...",         // vpn:// for AmneziaWG
  "proxy": {
    "port": 21080                             // required — proxy listen port
  },
  // ... existing fields unchanged
}
```

### vpn:// URI format

```
vpn://<base64url-encoded-json>
```

Decoded JSON structure (container type matched by checking if `"container"` field contains `"awg"`):
```json
{
  "containers": [{
    "container": "amnezia-awg",
    "awg": {
      "client_priv_key": "...",
      "client_pub_key": "...",
      "client_ip": "10.8.1.2/32",
      "server_pub_key": "...",
      "psk_key": "...",
      "config": "[Interface]\nPrivateKey=...\nAddress=...\n...",
      "hostName": "server.example.com",
      "port": 443,
      "Jc": "7", "Jmin": "150", "Jmax": "1000",
      "S1": "117", "S2": "321",
      "H1": "2008066467", "H2": "2351746464", "H3": "3053333659", "H4": "1789444460"
    }
  }]
}
```

### Xray config in AWG mode

The key difference: proxy outbound uses `freedom` protocol instead of a remote server protocol. This means traffic routed to "proxy" tag exits through the OS network stack, where AWG's routing table sends it through the tunnel.

```json
{
  "outbounds": [
    {"protocol": "freedom", "tag": "proxy", "settings": {}},
    {"protocol": "freedom", "tag": "direct"},
    {"protocol": "blackhole", "tag": "block"}
  ]
}
```

### EngineMode enum (simpler than trait for 2 engines)

```rust
pub enum EngineMode {
    Xray(ProxyParams),       // Full xray: tunnel + routing
    Awg(AwgConfig),          // AWG tunnel + xray routing layer
}
```

Match-based dispatch in `cmd_start`:
```rust
match engine_mode {
    EngineMode::Xray(params) => { /* existing flow */ },
    EngineMode::Awg(awg_config) => { /* AWG flow: start tunnel, then xray as router */ },
}
```

### Windows platform specifics

**Config paths:**
- Config: `%APPDATA%\corvex\corvex.json` (via `std::env::var("APPDATA")`)
- Xray config: `%APPDATA%\xray\config.json`
- Logs: `%LOCALAPPDATA%\corvex\corvex.log` (via `std::env::var("LOCALAPPDATA")`)

**Proxy control:**
- Registry: `HKCU\Software\Microsoft\Windows\CurrentVersion\Internet Settings`
  - `ProxyEnable` (DWORD): 0 = off, 1 = on
  - `ProxyServer` (String): `socks=127.0.0.1:21080`
- CLI: `netsh winhttp set proxy` / `netsh winhttp reset proxy`

**DNS discovery:**
- `GetAdaptersAddresses` → per-adapter DNS suffix + DNS servers
- Registry NRPT: `HKLM\SOFTWARE\Policies\Microsoft\Windows NT\DNSClient\DnsPolicyConfig\*`

**Process management:**
- Start: `std::process::Command` (cross-platform)
- Stop: `TerminateProcess` via `windows-sys`
- Reload: stop + start (no signal equivalent to SIGHUP)
- Is running: `OpenProcess` + `GetExitCodeProcess`

### AWG privilege requirements

`awg-quick up/down` requires root/sudo. Corvex will:
- Attempt `sudo awg-quick up <conf>` 
- Print clear error message if permission denied
- Document in help/README that AWG mode requires sudo
